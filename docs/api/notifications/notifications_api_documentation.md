# Notifications API Documentation

This module handles user notification preferences, scheduled reminders, and FCM push delivery.

**Base URL**: `/v1/notifications`  
**Authentication**: All endpoints require `Authorization: Bearer <access_token>`

> [!NOTE]
> Semua success response memakai envelope:
> `{ "Status": "...", "Message": "...", "Data": ... }`
>
> Semua error response memakai:
> `{ "Status": "...", "Message": "..." }`

Implementation notes:
- Push delivery now uses Firebase Admin SDK / ADC, not legacy `FCM_SERVER_KEY`.
- Scheduled reminders are generated by the notification worker service, not only by the API endpoint.
- For local or container runs, provide Firebase service-account credentials via `FIREBASE_CREDENTIALS_PATH`, `GOOGLE_APPLICATION_CREDENTIALS`, or `FIREBASE_CREDENTIALS_JSON`.

## What Mobile Needs To Prepare
This backend currently expects an **FCM device registration token** in `push_token`.

If the mobile app is built with Expo, the team should confirm whether they will:
- use native Firebase Cloud Messaging tokens, or
- stay with Expo Push Tokens and add a separate gateway later.

Current backend support is for **FCM tokens**, not Expo Push Tokens.

Mobile checklist:
- request notification permission from the user
- obtain a device FCM token after login or device registration
- send that token to `PATCH /v1/notifications/settings`
- refresh the token when Firebase rotates it
- handle foreground notifications manually in the app UI
- handle background notification taps and deep links
- show unread badges/counts using `GET /v1/notifications`
- mark notifications as read with `PATCH /v1/notifications/{id}/read`

Recommended mobile fields to store locally:
- `push_token`
- `notification_enabled`
- `daily_expense_reminder_enabled`
- `debt_payment_reminder_enabled`
- `salary_reminder_enabled`
- `daily_expense_reminder_time`
- `debt_payment_reminder_time`
- `salary_reminder_time`
- `salary_day`

## 1. Get Notification Settings
**GET** `/v1/notifications/settings`

Returns current user notification settings.

---

## 2. Update Notification Settings
**PATCH** `/v1/notifications/settings`

Controls enable/disable and custom schedule per user.

Use this endpoint after the app gets a push token. If `push_token` is empty, the backend will still save reminder history, but no push will be delivered to the device.

Request body example:
```json
{
  "enabled": true,
  "daily_expense_reminder_enabled": true,
  "daily_expense_reminder_time": "20:00",
  "debt_payment_reminder_enabled": true,
  "debt_payment_reminder_time": "09:00",
  "debt_payment_reminder_days_before": 3,
  "salary_reminder_enabled": true,
  "salary_reminder_time": "08:00",
  "salary_reminder_days_before": 1,
  "salary_day": 25,
  "push_token": "fcm-device-token"
}
```

## 3. Generate Scheduled Reminders
**POST** `/v1/notifications/generate`

Generates:
- daily expense input reminder
- debt payment reminder
- salary reminder

If `push_token` exists and Firebase credentials are configured, notifications are also pushed.
If you run the worker service in Docker Compose, this endpoint is called automatically on the cron schedule.

Note: `google-services.json` is for the mobile app. The backend needs a Firebase Admin service-account key.

Inbox rows are persisted in the backend and returned by `GET /v1/notifications`.
Each item includes `kind`, `type`, `read`, `read_at`, `created_at`, and `data` for routing.

Suggested notification types handled by the app:
- `daily_expense_input`
- `debt_payment`
- `salary_reminder`

Suggested UI behavior:
- `daily_expense_input`: open the quick expense input screen
- `debt_payment`: open debt detail or payment screen
- `salary_reminder`: open income / salary input screen

`salary_day` is the recurring day-of-month for salary. If the month is shorter than that day, the backend uses the last day of the month.

## 4. List Notifications
**GET** `/v1/notifications`

Optional query params:
- `kind=daily_expense_input`
- `read=true|false`

This endpoint is useful for:
- notification inbox
- unread badge count
- retrying missed taps
- showing reminder history even when push delivery failed

Response item example:
```json
{
  "id": 123,
  "kind": "daily_expense_input",
  "type": "daily_expense_input",
  "title": "Expense Reminder",
  "message": "Don’t forget to log today’s expenses.",
  "read": false,
  "read_at": null,
  "created_at": "2026-04-21T09:00:00Z",
  "data": {
    "kind": "daily_expense_input",
    "type": "daily_expense_input",
    "route": "\/activity"
  }
}
```

---

## 5. Mark Notification As Read
**PATCH** `/v1/notifications/{id}/read`

Marks the item as read and sets `read_at`.

Expected success response:
```json
{
  "Status": "success",
  "Message": "read",
  "Data": { "status": "read" }
}
```
